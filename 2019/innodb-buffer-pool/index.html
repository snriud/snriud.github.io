关于 InnoDB 存储引擎中的 buffer pool

众所周知，innodb 维护了一块内存空间，称作 buffer pool 用于缓存文件系统中读取的页。这里记录下我了解到的 innodb 对于这块内存空间的管理。

关于 buffer pool

buffer pool 可以有多个实例，也就是说可以有多块 buffer pool 来供系统使用，这个参数由 innodb_buffer_pool_instances 来控制。每个 page 的大小默认为 16K，由 innodb_page_size 这个参数可以查到。buffer pool 分配的内存在保证服务器其它进程够用的情况下，应尽可能大一些，以更好地利用合并写等特性来提高 IO 性能, 同样 buffer pool 的大小可以由 innodb_buffer_pool_size 来查看。

buffer pool 的维护

InnoDB 中使用两个链表来对 buffer pool 进行维护，一个链表记录已读入内存中的页(database pages)，另一个链表来记录尚未使用的页(free buffers), 记录已读入内存中页的链表又分为两部分，前面一部分称为 young list, 后面一部分称为 old list, 参数 innodb_old_blocks_pct 记录他们之间的界限，默认值为 3/8, 即 young list 占用 5/8, old list 占用 3/8, 这个分界的位置也叫做 midpoint；这些链表中的页使用 LRU (Least Recently Used) 算法来进行管理。

当客户端执行一些 SQL 读操作时，将会首先去 old list 中寻找这个页，如果未找到，则从 free buffers 维护的链表中取一个页来存储从磁盘中读到的页，这个页被加入到 old list 的头部。 当 free buffers 中没有多余的空闲页可供使用时，会将 old list 尾部中的 page 进行驱逐，然后存放新读入的页，置于 old list 头部。

如果要查询的页最初就在 old list 中，则将这个页加入到 young list 中，这个操作也叫作 pages made young。

很容易想到，某一个 SQL 查询的结果集如果非常大（对应于业务中的生成报表等业务）, old list / young list 会很快被替换掉，这样，之前的所有缓存都将失效，如果我们再次访问时，需要重新进行 IO 操作，读入缓存中。这种情况是我们不愿意看到的。为此，InnoDB 引入了 innodb_old_blocks_time 来控制 old list 中的页必须等待多久之后才会在再次访问时被加入到 young list 中。这个参数的默认值为 1000，单位为毫秒。并且，这个值是全局动态的，因此，我们在执行此类操作时，可以先将这个参数调整到一个比较大的值，待操作完成后，再调回原值，避免将原来的热数据淘汰掉。

其中：

  • Buffer pool size 是指分配给 buffer pool 的页的数量
  • Free buffers 指尚未使用的页数量
  • Database pages 指已使用的页数量，young list / old list 中使用的数量
  • Modified db pages 指脏页数量
  • Old database pages 指 old list 中页的数量
  • Pages make young, not young 指被从 old list 移动到 young list 中的次数，以及反向操作的次数
  • youngs/s, no-youngs/s 每秒这两类操作的次数

Free buffers 与 Database pages 之和可能会小于 buffer pool size ，因为缓冲池中的页还会被分配给自适应哈希索引，Insert Buffer 等。
当 Database pages 中的页被修改后，这个页就会同磁盘上的版本不一致，因此变为了脏页，这个脏页会在 Database pages 以及 Modified db pages 中维护，等待 checkpoint 机制刷新回磁盘。
